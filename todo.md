## 1. Codex Goal

현재 자동 1v1 전투 프로토타입을 다음 MVP 게임으로 확장한다.

> 1세대 151마리 포켓몬을 도감 기반으로 언락하고, 스타터 선택 -> 상점/팀정비 -> 자동 전투 -> 포획/트레이너전 결과 처리 루프를 `localStorage`와 백그라운드 동기화 큐 기반으로 구현한다.

성공 기준:

- 도감 스타터 선택 씬, 상점/팀정비 씬, 전투 씬이 실제 플레이 흐름으로 연결된다.
- 플레이어 팀은 최대 6슬롯이며 모바일 기본 UI는 2열 x 3행이다.
- 야생전은 승리 후 포켓볼 1회 포획 시도를 제공한다.
- 트레이너전은 5웨이브 단위로 발생하고, 50웨이브 이후에는 50웨이브 트레이너전을 반복한다.
- 5-50웨이브 트레이너전 승리 시에만 팀 HP를 100% 회복한다.
- 50웨이브 이후 반복 구간은 HP 자동 회복과 코인 보상을 제공하지 않는다.
- 패배 시 별도 게임오버 화면 없이 전투 씬 상단에서 필요한 결과 편집을 끝낸 뒤 스타터 선택 도감으로 이동한다.
- 트레이너전 스냅샷/결과 전송 실패는 게임 흐름을 막지 않고 `apt.syncQueue.v1`에 재시도 작업으로 남긴다.
- `npm run test`와 `npm run build`가 통과한다.

## 3. 확정된 설계 결정

- 콘텐츠 범위는 Gen1 151마리와 Gen1 기술/타입이다.
- 내부 전투 스탯은 현재 구현처럼 `hp`, `atk`, `def`, `spa`, `spd`, `spe`를 사용한다.
- 모바일 팀 슬롯 기본 레이아웃은 2열 x 3행이다.
- 매칭은 같은 웨이브 안에서 같은 `playerId`를 제외한 랜덤 선택이다.
- 포켓볼 보유 수량은 모험 시작 시 `apt.run.v1`에 생성하고, 모험 종료 시 다음 런에서 리셋한다.
- 새 모험 기본 포켓볼은 Poke Ball 3개다.
- 포켓볼 상품 상세는 현재 보유 수량과 포켓볼 설명을 우선 표시한다.
- 트레이너전 패배 시 인사말 입력은 생략하고 팀 이름만 편집한다.
- 트레이너전 승리 시 팀 이름과 20자 이내 인사말을 편집한다.

## 4. 유지보수 구조 원칙

### 모듈 경계

- `core/`는 순수 게임 규칙, 타입, 계산, 상태 전이를 담당한다.
- `app/`은 앱 상태 머신, 이벤트 라우팅, 저장/동기화 호출 조립을 담당한다.
- `ui/`는 상태를 HTML 문자열로 렌더링하고 `data-action` 기반 이벤트만 노출한다.
- `styles/`는 씬/컴포넌트별 CSS를 분리한다.
- 저장소 접근, API 호출, 랜덤 생성은 어댑터로 격리하고 UI 컴포넌트에서 직접 호출하지 않는다.

기존 파일명을 유지하는 편이 더 안전하면 위 구조를 그대로 강제하지 않는다. 다만 책임 경계는 반드시 유지한다.

### 컴포넌트 규칙

- 렌더 함수는 가능한 한 `props -> string` 순수 함수로 유지한다.
- 이벤트 처리는 `data-action`, `data-id`, `data-slot` 같은 안정적인 속성으로 위임한다.
- 팀 패널은 상점/팀정비와 전투 씬에서 같은 컴포넌트를 재사용한다.
- 상점 패널은 구매 모드와 포획 아이템 사용 모드에서 같은 리스트/상세 컴포넌트를 재사용한다.
- 도감 카드, 팀 슬롯 카드, 상품 행은 각각 단일 컴포넌트에서 상태 변형만 처리한다.
- HTML에 사용자 입력을 넣을 때는 항상 `escapeHtml`을 사용한다.
- 타입은 먼저 만들고, 상태 전이 함수가 그 타입을 소비하게 한다.

## 5. 금지 사항

- 새 프레임워크를 도입하지 않는다. React, Vue, Svelte, Redux, Router 도입 금지.
- UI 컴포넌트에서 `localStorage`, `fetch`, Apps Script URL을 직접 호출하지 않는다.
- 백엔드 동기화 성공을 기다리느라 화면 전환을 막지 않는다.
- 트레이너전 매칭에 `teamPower` 필터나 보정을 넣지 않는다.
- 50웨이브 이후 반복 구간에 코인 보상이나 HP 자동 회복을 넣지 않는다.
- 패배 후 별도 게임오버 화면을 추가하지 않는다.
- 팀 패널과 상점 패널 마크업을 씬마다 복붙하지 않는다.
- 포켓몬 151마리 목록을 수동 하드코딩하지 않는다. 덱 데이터와 이미지 경로 헬퍼에서 파생한다.
- 상태 전이를 DOM 상태에서 역산하지 않는다. 앱 상태가 단일 출처여야 한다.
- 전투/포획/상점 계산을 UI 문자열 안에 섞지 않는다.
- 테스트 없이 저장 스키마나 전투 상태 구조를 바꾸지 않는다.
- 사용자 입력을 `innerHTML`에 그대로 넣지 않는다.
- 외부 이미지/CDN에 의존하지 않는다. 기존 정적 자산을 사용한다.
- 전체 앱을 한 파일에 몰아넣지 않는다.
- unrelated refactor, 대규모 스타일 재작성, 의미 없는 파일 이동을 하지 않는다.

- [ ] 포획 실패 시 상점/팀정비 씬으로 이동한다.

### 6.8 트레이너전 스냅샷과 동기화

- [ ] `teamJson`은 6슬롯, 확정 스탯, 레벨, 스킬, `pokemonPower`를 포함한다.
- [ ] `teamJson`에는 현재 HP, 버프, 랭크 변화, 이상 상태를 저장하지 않는다.
- [ ] 상대 스냅샷을 전투에 불러올 때 HP는 항상 `maxHp`로 시작한다.
- [ ] 같은 웨이브에서 같은 `playerId`를 제외하고 랜덤 상대를 요청한다.
- [ ] 상대 후보가 없으면 데이터 준비 오류로 처리하고 전투를 시작하지 않는다.
- [ ] 트레이너전 결과가 나오면 스냅샷 저장 작업과 결과 기록 작업을 큐에 등록한다.
- [ ] 동기화 성공 시 큐 작업을 제거한다.
- [ ] 동기화 실패 시 큐 작업을 남기고 화면 전환은 계속 진행한다.
- [ ] `sendBeacon` 또는 동등한 방식으로 종료 직전 전송을 시도한다.
- [ ] 6시간 보상 조회 실패도 게임 흐름을 막지 않고 재시도 대상으로 남긴다.

### 6.9 전투 씬 UI

- [ ] 기존 `battle-frame` 구조와 클래스명을 유지한다.
- [ ] 상단 전투 HUD, 스프라이트 슬롯, 메시지 박스는 기존 원작풍 패널을 재사용한다.
- [ ] 하단에는 전투 중 읽기 전용 팀 패널을 표시한다.
- [ ] 현재 출전 슬롯과 기절 슬롯을 시각적으로 구분한다.
- [ ] 야생전 승리 후 하단을 포켓볼 사용 탭으로 전환한다.
- [ ] 트레이너전 결과 후 포획 탭을 표시하지 않는다.
- [ ] 트레이너전 승리 시 축하 꽃가루 연출을 표시한다.
- [ ] 트레이너전 승리 시 팀 이름과 인사말 편집을 표시한다.
- [ ] 트레이너전 패배 시 팀 이름 편집만 표시한다.
- [ ] 저장된 팀 이름과 인사말을 결과 편집 기본값으로 자동 로드한다.
- [ ] 결과 편집 완료 후 승리면 상점/팀정비, 패배면 도감 스타터 선택으로 이동한다.

## 7. 구현 순서 제안

1. 타입과 저장 계층을 먼저 만든다.
2. 도감 스타터 선택과 새 모험 시작을 연결한다.
3. 6슬롯 팀 패널을 만들고 상점/전투에서 재사용한다.
4. 상점 상품과 대상 선택 UX를 구현한다.
5. 기존 전투 엔진을 6슬롯 팀 전투 루프로 감싼다.
6. 야생전 포획 플로우를 붙인다.
7. 트레이너전 매칭, 결과 편집, 동기화 큐를 붙인다.
8. 50웨이브 반복 구간과 보상 예외를 검증한다.
9. 반응형 UI와 테스트를 마감한다.

## 8. Definition of Done

- 유저가 첫 접속에서 도감을 보고 스타터를 선택할 수 있다.
- 스타터 선택 후 상점/팀정비 씬으로 이동한다.
- 상점에서 팀 상태, 재화, 상품 상세, 대상 선택형 상품이 동작한다.
- 전투 시작 후 야생전과 트레이너전이 웨이브 규칙에 따라 발생한다.
- 야생전 승리 후 포켓볼 포획 플로우가 동작한다.
- 포획 성공 포켓몬은 빈 슬롯 자동 합류 또는 무료 대기 상품으로 처리된다.
- 트레이너전 승리/패배 결과 편집 흐름이 설계대로 분기한다.
- 모든 진행 상태는 지정된 `localStorage` key로 복구 가능하다.
- 백엔드 동기화 실패는 유저 흐름을 막지 않고 재시도 큐에 남는다.
- 모바일 360px에서 주요 UI가 겹치지 않는다.
- `npm run test`와 `npm run build`가 통과한다.
