# Async Pocket Trainer 게임 설계 문서

## 1. 요약

`Async Pocket Trainer`는 1세대 포켓몬 151마리를 수집하고, 6개 팀 슬롯을 계속 교체하며, 자동 전투로 웨이브를 돌파하는 모바일 웹 게임이다. 핵심 방향은 `Backpack Battles`처럼 실시간 PvP가 아니라 다른 유저가 남긴 팀 스냅샷과 비동기로 싸우는 것이다.

- 클라이언트는 GitHub Pages 배포 가능한 Vite + TypeScript 정적 웹앱으로 유지한다.
- 포켓몬과 전투 규칙은 1세대 기준으로 시작하고, 기존 `src/resources/pokemon/0001.webp`-`0151.webp` 이미지를 사용한다.
- 전투 엔진은 Pokemon Showdown의 Gen1 데이터/로직을 참고한 로컬 경량 구현을 유지한다.
- Google Sheets를 MVP용 비동기 상대 풀 저장소로 사용하고, Google Apps Script 웹앱을 얇은 백엔드 API로 둔다.
- MVP는 웨이브 진행, 야생 포켓몬 포획, 보유 포켓몬 비교/교체, 6슬롯 팀 운용, 5웨이브 단위 비동기 트레이너전을 포함한다.

## 2. 핵심 재미

### 포켓몬을 아이템처럼 많이 얻는 게임

포켓몬은 희귀한 단일 캐릭터가 아니라 디아블로식 장비처럼 자주 얻고 비교하는 대상으로 설계한다. 플레이어는 새 포켓몬을 얻을 때마다 현재 팀과 비교하고, 6개 슬롯 중 무엇을 유지하거나 교체할지 빠르게 판단한다.

- 비교 기준: 종족, 타입, 레벨, HP, 공격/방어/특수/스피드, 공격기, 보조기, 현재 상태.
- 교체 판단: 즉시 팀 슬롯에 넣기, 보유 목록에 보관하기, 버리기 중 하나를 선택한다.
- 플레이 감각: 좋은 포켓몬이 계속 떨어지고, 팀 조합이 웨이브마다 갱신되는 수집형 자동전투.

### 비동기 유저 팀과 싸우는 게임

플레이어는 일정 웨이브마다 야생 포켓몬이 아니라 다른 유저의 과거 팀 조합과 싸운다. 상대는 실시간 접속자가 아니라 Google Sheets에 저장된 팀 스냅샷이다.

- 5, 10, 15... 웨이브는 트레이너 배틀 웨이브다.
- 트레이너 배틀 상대는 같은 웨이브 구간에서 다른 유저가 사용했던 6슬롯 팀 상태다.
- 유저가 모험을 종료하거나 체크포인트에 도달하면 현재 팀 상태를 백엔드에 업로드한다.
- 다른 유저의 스냅샷은 이후 플레이어의 트레이너 배틀 콘텐츠가 된다.

## 3. MVP 게임 루프

1. 플레이어가 새 모험을 시작한다.
2. 게임은 시드, `playerId`, `runId`를 생성하고 초기 팀을 구성한다.
3. 일반 웨이브에서는 야생 포켓몬 1마리와 자동 전투한다.
4. 야생 포켓몬의 HP가 0이 되면 포켓볼을 1회 던져 포획을 시도한다.
5. 포획 성공 시 보유 목록에 추가하고, 플레이어는 6개 팀 슬롯 중 교체 여부를 선택한다.
6. 5의 배수 웨이브에서는 야생 포켓몬 대신 비동기 트레이너 팀 스냅샷과 자동 전투한다.
7. 트레이너 배틀이 끝나면 결과와 현재 팀 스냅샷을 Google Apps Script API로 전송한다.
8. 패배하거나 플레이어가 모험을 떠나면 마지막 팀 상태를 저장하고 런을 종료한다.

## 4. 웨이브 설계

- 웨이브 번호는 1부터 시작한다.
- 일반 웨이브: `wave % 5 !== 0`
  - 야생 포켓몬 1마리가 등장한다.
  - 야생 포켓몬은 현재 웨이브를 기준으로 레벨 범위를 점진 상승시킨다.
  - 승리 후 포획 기회를 1회 제공한다.
- 트레이너 배틀 웨이브: `wave % 5 === 0`
  - 야생 포켓몬은 등장하지 않는다.
  - Apps Script API에서 해당 웨이브의 상대 팀 스냅샷을 요청한다.
  - 상대 스냅샷이 없거나 API가 실패하면 로컬 기본 트레이너 팀을 사용한다.

MVP 레벨 스케일링 기본값:

- 1-4웨이브: Lv.5-10
- 6-9웨이브: Lv.8-14
- 11-14웨이브: Lv.12-18
- 이후 5웨이브마다 최소/최대 레벨을 3씩 올린다.
- 레벨 상한은 MVP에서 Lv.50으로 제한한다.

## 5. 전투 설계

### 기본 전투

- 자동 전투이며 플레이어는 전투 중 기술을 직접 선택하지 않는다.
- 현재 구현된 Gen1 기반 로컬 전투 엔진을 확장한다.
- MVP 전투는 PP를 소모하지 않는다.
- 기술 선택은 기존 방향을 유지한다.
  - 공격기 선택 확률 70%
  - 보조기 선택 확률 30%
  - 각 포켓몬은 레벨 이하에서 배운 공격기 1개와 보조기 1개를 시드 기반으로 보유한다.
  - 조건을 만족하는 기술이 없으면 타입별 기본 공격기와 공용 보조기 fallback을 사용한다.

### 팀 전투

MVP 최종 전투 단위는 1v1 단판이 아니라 6슬롯 팀 전투다.

- 각 팀은 최대 6마리 포켓몬을 가진다.
- 전투 시작 시 각 팀의 첫 번째 생존 포켓몬이 출전한다.
- 포켓몬이 쓰러지면 다음 생존 포켓몬이 자동 출전한다.
- 한쪽 팀의 모든 포켓몬이 쓰러지면 전투가 종료된다.
- 현재 1v1 엔진은 팀 전투의 단일 매치업 해석기로 사용한다.

### 포획

- 포획은 야생 포켓몬 전투에서만 발생한다.
- 야생 포켓몬 HP가 0이 된 뒤 포켓볼을 1회만 던진다.
- 트레이너 배틀에서는 포획이 발생하지 않는다.
- MVP 포획률은 단순 공식으로 시작한다.
  - 기본 포획률 50%
  - 플레이어 팀 평균 레벨보다 야생 포켓몬 레벨이 낮으면 +10%
  - 야생 포켓몬이 최종 진화형 또는 희귀 포켓몬이면 -15%
  - 최종 확률은 10%-90%로 제한한다.
- 포획 성공/실패는 시드 기반 RNG로 재현 가능해야 한다.

## 6. 팀과 보유 포켓몬 설계

### 팀 슬롯

- 플레이어 팀은 6개 고정 슬롯을 가진다.
- 빈 슬롯이 있으면 포획 성공 포켓몬을 자동으로 가장 앞의 빈 슬롯에 배치한다.
- 팀이 가득 찼으면 포획 성공 후 교체 화면을 보여준다.
- 교체 화면은 포획 포켓몬과 현재 6마리를 한 화면에서 비교하게 한다.

### 보유 목록

- MVP에서는 보유 목록을 런 내부 임시 인벤토리로 둔다.
- 모험이 끝나면 영구 컬렉션 저장은 하지 않는다.
- 장기 확장에서는 영구 보관함, 도감, 강화, 진화, 해체 보상 등을 추가할 수 있다.

### 비교 UI

포켓몬 카드에는 다음 정보를 항상 표시한다.

- 이름, 도감 번호, 타입, 레벨
- 현재 HP/최대 HP
- 주요 스탯
- 공격기/보조기
- 팀 슬롯 번호 또는 보유 상태

## 7. 비동기 스냅샷 동기화 설계

### 백엔드 구조

- Google Apps Script를 웹앱으로 배포한다.
- Google Sheets는 MVP 데이터 저장소로 사용한다.
- 클라이언트는 Apps Script URL에 HTTP 요청을 보내고, Apps Script가 Sheets를 읽고 쓴다.
- 클라이언트에는 Google Sheets API 키나 OAuth 비밀 정보를 넣지 않는다.

### 저장 시점

클라이언트는 다음 시점에 팀 스냅샷을 전송한다.

- 새 모험 시작 후 초기 팀 확정 시
- 5의 배수 웨이브 트레이너 배틀 종료 시
- 플레이어가 모험을 떠나거나 패배해 런이 종료될 때
- 앱이 닫히기 직전 가능한 경우 `sendBeacon` 또는 동일 목적의 비동기 전송 사용

### Sheets 컬럼

`team_snapshots` 시트는 다음 컬럼을 가진다.

| 컬럼 | 설명 |
| --- | --- |
| `snapshotId` | 스냅샷 고유 ID |
| `playerId` | 로컬 생성 유저 ID |
| `runId` | 모험 1회를 구분하는 ID |
| `wave` | 저장 시점의 웨이브 번호 |
| `seed` | 런 시드 |
| `teamJson` | 6슬롯 팀 상태 JSON |
| `resultSummary` | 직전 전투 결과 요약 |
| `createdAt` | ISO 문자열 생성 시각 |

### 팀 스냅샷 JSON

`teamJson`은 다음 형태를 기본 계약으로 사용한다.

```json
{
  "slots": [
    {
      "slot": 1,
      "speciesId": "pikachu",
      "level": 12,
      "hp": 31,
      "maxHp": 35,
      "selectedMoves": {
        "attack": "thundershock",
        "support": "growl"
      },
      "status": null
    }
  ]
}
```

### API 계약

Apps Script 웹앱은 쿼리 파라미터 `action`으로 동작을 구분한다.

#### 스냅샷 저장

`POST {APPS_SCRIPT_URL}?action=submitSnapshot`

요청:

```json
{
  "playerId": "local-player-id",
  "runId": "run-id",
  "wave": 10,
  "seed": "battle-seed",
  "team": {
    "slots": []
  },
  "resultSummary": {
    "type": "trainer",
    "won": true
  }
}
```

응답:

```json
{
  "ok": true,
  "snapshotId": "snapshot-id"
}
```

#### 상대 스냅샷 조회

`GET {APPS_SCRIPT_URL}?action=matchSnapshot&wave=10&excludePlayerId=local-player-id&seed=battle-seed`

응답:

```json
{
  "ok": true,
  "snapshot": {
    "snapshotId": "snapshot-id",
    "playerId": "opponent-player-id",
    "wave": 10,
    "team": {
      "slots": []
    }
  }
}
```

상대가 없을 때:

```json
{
  "ok": false,
  "reason": "NO_MATCH"
}
```

### 매칭 규칙

- 같은 `playerId`의 스냅샷은 상대 후보에서 제외한다.
- 우선 같은 웨이브 번호의 스냅샷을 찾는다.
- 같은 웨이브 후보가 없으면 `wave - 5`부터 `wave + 5` 범위에서 가장 가까운 후보를 찾는다.
- 후보가 여러 개면 요청 시드 기반으로 하나를 선택한다.
- 그래도 없으면 로컬 fallback 트레이너를 사용한다.

## 8. UI 설계

### 기본 화면

- 상단: 현재 웨이브, 런 시드, 팀 전투 상태, 자동 진행/일시정지 버튼.
- 중앙: 플레이어 팀 6슬롯과 현재 상대.
- 하단: 전투 로그, 포획 결과, 교체 선택.

### 주요 플로우

- 일반 웨이브 승리 후:
  - 포켓볼 1회 연출
  - 성공/실패 결과 표시
  - 성공 시 팀 교체 UI 표시
- 트레이너 배틀 진입 전:
  - 상대 트레이너 카드 표시
  - 상대가 실제 다른 유저 스냅샷인지 fallback인지 표시
- 팀 교체:
  - 현재 6슬롯과 새 포켓몬을 한 화면에 표시
  - 교체할 슬롯 선택 또는 보관/버리기 선택

## 9. 테스트 계획

- 데이터 검증
  - 1세대 151마리 이미지가 존재해야 한다.
  - 모든 포켓몬은 공격기/보조기 선택 또는 fallback이 가능해야 한다.
- 전투 검증
  - 같은 시드와 같은 팀 스냅샷은 같은 전투 로그와 결과를 만든다.
  - 6슬롯 팀 전투에서 쓰러진 포켓몬 다음 슬롯이 자동 출전한다.
  - 한쪽 팀이 모두 쓰러지면 승패가 확정된다.
- 웨이브 검증
  - `wave % 5 !== 0`에서는 야생 포켓몬 전투가 발생한다.
  - `wave % 5 === 0`에서는 트레이너 스냅샷 전투가 발생한다.
  - 트레이너 스냅샷 조회 실패 시 fallback 상대가 생성된다.
- 포획 검증
  - 야생 포켓몬 HP가 0이 된 뒤 1회만 포획 판정이 발생한다.
  - 트레이너 배틀에서는 포획이 발생하지 않는다.
  - 같은 시드는 같은 포획 결과를 만든다.
- 동기화 검증
  - Apps Script 저장 API가 Sheets에 스냅샷을 추가한다.
  - 매칭 API가 같은 `playerId`를 제외하고 상대 스냅샷을 반환한다.
  - API 실패 시 게임 진행이 막히지 않는다.
- UI 검증
  - 모바일 폭 360px에서 6슬롯 팀, 전투 로그, 포획/교체 UI가 겹치지 않는다.
  - GitHub Pages base path에서 이미지와 정적 리소스가 로드된다.
- 빌드 검증
  - `npm run test`가 통과해야 한다.
  - `npm run build`가 통과해야 한다.

## 10. 후속 확장

MVP 이후에 다음 시스템을 확장 후보로 둔다.

- 영구 보관함과 도감 진행도
- 포켓몬 진화와 레벨업
- 아이템, 장비, 특성, 룬 같은 빌드 변형 요소
- 포켓몬 해체/판매/강화 재료화
- 시즌별 Google Sheets 분리와 랭킹
- 더 정교한 상대 매칭, 난이도 보정, 메타 통계

## 11. 가정과 라이선스

- UI 언어는 한국어로 한다.
- MVP 포켓몬 범위는 1세대 151마리로 제한한다.
- 런타임 전투 엔진 패키지 의존성은 두지 않고, 필요한 Gen1 로직만 로컬 구현으로 유지한다.
- Google Sheets는 장기 운영 DB가 아니라 MVP 검증용 저장소다.
- Pokemon Showdown의 MIT 라이선스와 출처는 `NOTICE` 또는 앱 내 크레딧에 명시한다.
- 참고 소스는 Smogon Pokemon Showdown 공식 저장소의 MIT 코드와 데이터다: https://github.com/smogon/pokemon-showdown
