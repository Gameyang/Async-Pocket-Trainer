# Async Pocket Trainer 코드 개발 하드닝 엔지니어링 가이드

## 1. 목적

이 문서는 `todo.md`의 MVP 구현을 코드 품질, 유지보수성, 테스트 안정성, 프론트엔드 보안 관점에서 단단하게 만들기 위한 개발 기준이다. 기능 구현 속도보다 중요한 것은 다음 네 가지다.

- 게임 상태가 한 곳에서 설명 가능해야 한다.
- 화면은 같은 상태를 항상 같은 방식으로 렌더링해야 한다.
- 저장/동기화 실패가 유저 플레이 흐름을 깨지 않아야 한다.
- 다음 LLM이나 개발자가 파일 몇 개만 읽고도 안전하게 수정할 수 있어야 한다.

## 2. 참고한 GitHub 레퍼런스

아래 공개 GitHub 자료의 원칙을 이 프로젝트에 맞게 축약했다.

- `labs42io/clean-code-typescript`: TypeScript에서 읽기 쉽고 재사용 가능하며 리팩터링 가능한 코드, SOLID, 테스트, 오류 처리, 주석 절제를 강조한다.
- `goldbergyoni/javascript-testing-best-practices`: 테스트는 요구사항 수준에서 읽혀야 하고, 현실적인 입력과 공개 동작 중심 검증을 우선한다.
- `thedaviddias/Front-End-Checklist`: 프론트엔드 보안, 성능, 접근성, HTML/CSS/JS 검증 항목을 넓게 다룬다.
- `mercari/production-readiness-checklist`: 개발 전 설계 체크와 출시 전 체크를 나눠 준비 상태를 검토한다.
- `github/well-architected` Application Security Checklist: 의존성 스캔, 코드 스캔, 시크릿 관리, 모듈성, 저장소 구조 점검을 강조한다.

이 프로젝트는 정적 Vite 게임 앱이므로 서버 운영 체크리스트를 그대로 적용하지 않는다. 대신 상태 저장, 동기화 큐, 빌드/테스트, 브라우저 보안, UI 회귀 방지에 집중한다.

## 3. 하드닝 원칙

### 3.1 상태 단일 출처

- 앱 상태는 TypeScript 객체가 단일 출처다.
- DOM 클래스, 버튼 텍스트, HTML 문자열에서 게임 상태를 역산하지 않는다.
- 모든 씬 전환은 명시적인 `SceneState`와 `Action`으로만 발생한다.
- `localStorage` 저장 전후에는 런타임 검증 또는 복구 함수를 반드시 통과한다.
- 저장 key version이 맞지 않으면 조용히 깨진 상태로 진행하지 말고 기본값 복구 경로를 탄다.

### 3.2 모듈 경계

- `core/`는 DOM, `localStorage`, `fetch`를 몰라야 한다.
- `ui/`는 계산하지 않고 렌더링만 한다.
- `app/`은 입력 이벤트를 받아 `core` 상태 전이를 호출하고 다시 렌더링한다.
- 외부 API와 저장소는 어댑터로 감싼다.
- 새 기능은 가능한 한 `type -> pure function -> render` 순서로 만든다.

### 3.3 타입 우선

- 새 상태 필드는 먼저 타입과 기본값을 만든다.
- `any`는 금지한다. 외부 JSON 파싱 직후처럼 불가피한 지점은 `unknown`으로 받고 검증 후 좁힌다.
- `PokemonInstance`, `RunState`, `ShopProduct`, `SyncQueueItem`처럼 도메인 이름이 드러나는 타입을 사용한다.
- 문자열 리터럴 상태는 union 타입으로 고정한다.
- `null`과 빈 슬롯은 의미가 분명한 곳에서만 사용한다.

### 3.4 UI 재사용

- 팀 패널은 상점/팀정비와 전투 씬에서 같은 컴포넌트를 재사용한다.
- 상점 패널은 구매 모드와 포획 아이템 사용 모드에서 같은 리스트/상세 구조를 재사용한다.
- 도감 카드, 팀 슬롯 카드, 상품 행은 상태 변형만 다르게 렌더링한다.
- 씬마다 비슷한 HTML을 복붙하지 않는다.
- CSS 클래스는 역할 기반으로 붙인다. 시각 효과 이름만 있는 클래스는 상태 클래스와 분리한다.

### 3.5 실패 허용 설계

- Google Apps Script 요청 실패는 플레이 흐름을 막지 않는다.
- 실패한 스냅샷 저장, 결과 기록, 보상 조회는 `apt.syncQueue.v1`에 남긴다.
- 큐 작업은 중복 전송을 고려해 `id`, `kind`, `createdAt`, `attempts`, `payload`를 가진다.
- 네트워크 실패와 서버 실패는 UI에서 구분하지 않아도 되지만, 큐에는 재시도 가능한 형태로 남긴다.
- 동기화 큐가 깨져도 런 진행 상태는 복구 가능해야 한다.

### 3.6 보안 기본선

- 사용자 입력은 항상 `escapeHtml` 후 렌더링한다.
- `innerHTML`에 사용자 입력 원문을 직접 넣지 않는다.
- Apps Script URL, 토큰, 시크릿은 커밋하지 않는다.
- 외부 CDN, 외부 이미지, 원격 스크립트를 새로 추가하지 않는다.
- 클릭 가능한 액션은 `data-action` 화이트리스트로 처리한다.
- 저장된 인사말, 팀 이름, 트레이너 이름은 길이 제한과 제어문자 제거를 거친다.

### 3.7 성능과 모바일 안정성

- 151개 도감 카드는 이미지 크기와 카드 크기를 고정해 레이아웃 흔들림을 줄인다.
- 포켓몬 이미지는 기존 WebP 자산을 사용한다.
- 모바일 360px에서 텍스트가 버튼/카드 밖으로 넘치지 않아야 한다.
- 전투 중 애니메이션은 상태 전환을 방해하지 않아야 한다.
- 한 화면에서 주 스크롤 영역은 하나로 제한한다.
- 이미지, HP 바, 슬롯 카드처럼 반복되는 UI는 고정 치수 또는 반응형 제약을 둔다.

## 4. 기능별 하드닝 게이트

### 4.1 도감 스타터 선택

- [ ] 151종 목록은 덱 데이터에서 파생한다.
- [ ] 잠금/언락 판정은 `apt.dex.v1`만 본다.
- [ ] 선택 중 스크롤과 배경 클릭이 차단된다.
- [ ] 잠긴 카드 클릭은 상태를 바꾸지 않는다.
- [ ] 스타터 생성은 렌더 함수가 아니라 상태 전이 함수에서 한다.

### 4.2 상점/팀정비

- [ ] 상품 구매 전 재화와 사용처를 모두 검증한다.
- [ ] 대상 선택형 상품은 적용 가능 슬롯 목록을 상태로 가진다.
- [ ] 취소 시 환불은 단일 함수에서 처리한다.
- [ ] 포켓몬 판매와 회복은 같은 슬롯 선택 구조를 공유한다.
- [ ] 포켓볼 상품은 구매 시 수량 +1, 포획 사용 시 수량 -1 규칙을 각각 테스트한다.

### 4.3 전투와 포획

- [ ] 6슬롯 팀 전투는 기존 1v1 엔진을 감싸는 계층에서 처리한다.
- [ ] 쓰러진 뒤 다음 생존 슬롯 선택은 순수 함수로 테스트한다.
- [ ] 야생전 포획은 1회만 가능하다.
- [ ] 포획 성공 포켓몬은 HP 50% 상태로 생성된다.
- [ ] 빈 슬롯이 없으면 무료 대기 상품으로만 이동한다.

### 4.4 트레이너전과 동기화

- [ ] `teamJson`은 현재 HP와 전투 중 일시 상태를 저장하지 않는다.
- [ ] 상대 스냅샷 전투 시작 HP는 항상 `maxHp`다.
- [ ] 매칭은 같은 웨이브 랜덤이며 `teamPower`를 필터로 쓰지 않는다.
- [ ] 결과 화면 전환은 동기화 성공을 기다리지 않는다.
- [ ] 패배 시 팀 이름만 편집하고 인사말 입력은 생략한다.

## 5. 테스트 하드닝

### 필수 테스트 유형

- 순수 규칙 테스트: 웨이브, 보상, 포획률, 슬롯 선택, 회복, 판매.
- 저장 복구 테스트: key 없음, JSON 파싱 실패, version 불일치, 필수 필드 누락.
- 상태 전이 테스트: 도감 -> 상점 -> 전투 -> 포획/결과 -> 다음 씬.
- 동기화 큐 테스트: 성공 제거, 실패 유지, 중복 방지.
- 렌더링 스냅샷 또는 DOM 문자열 테스트: 주요 버튼과 `data-action` 존재.

### 테스트 작성 규칙

- 테스트 이름은 조건과 기대 결과를 드러내야 한다.
- 구현 내부보다 공개 함수와 상태 결과를 검증한다.
- 랜덤이 필요한 테스트는 seed 가능한 RNG를 주입한다.
- 날짜/시간은 직접 호출하지 말고 clock 값을 주입한다.
- 테스트 입력은 `foo`, `bar`보다 실제 도메인 예시를 쓴다.

## 6. PR/작업 완료 체크

각 작업은 완료 전에 아래를 확인한다.

- [ ] `todo.md`의 해당 체크 항목을 만족한다.
- [ ] 새 타입과 상태 전이가 문서의 용어와 일치한다.
- [ ] UI 컴포넌트가 저장소/API에 직접 접근하지 않는다.
- [ ] 사용자 입력은 escape/sanitize 경로를 지난다.
- [ ] 실패 가능한 네트워크 작업은 큐 또는 명시적 실패 상태를 가진다.
- [ ] 모바일 360px에서 주요 텍스트와 버튼이 겹치지 않는다.
- [ ] 신규 규칙에는 테스트가 있다.
- [ ] `npm run test`를 통과한다.
- [ ] `npm run build`를 통과한다.

## 7. Codex 작업 규칙

- 작업 시작 전 `todo.md`, `docs/game-direction-plan.md`, `docs/game-visual-plan.md`, 이 문서를 읽는다.
- 먼저 작은 타입/순수 함수 단위로 구현하고, 그 다음 앱 상태와 UI에 연결한다.
- 큰 기능을 한 번에 끝내려 하지 말고 저장, 도감, 팀, 상점, 전투, 동기화 순서로 나눈다.
- 기존 `battle-frame` 시각 구조는 유지한다.
- 새 패키지는 꼭 필요할 때만 추가한다. 추가 전 이유와 대체안을 코드 리뷰에서 설명할 수 있어야 한다.
- 리팩터링은 현재 기능 구현에 필요한 범위로 제한한다.
- 완료 보고에는 변경 파일, 검증 명령, 남은 위험을 짧게 적는다.
